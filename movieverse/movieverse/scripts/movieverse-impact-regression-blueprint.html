Problem definition

Three problems for the MovieVerse codebase

P1

Index and understand the code

Build a semantic + structural index of all 6 domain services so any developer or agent can query the codebase by meaning — not by grepping files one by one.

P2

Find regression areas on change

When a developer changes the TicketingService, MovieRepository, or any shared class, automatically find every other service, test, and API endpoint that may regress.

P3

Fix impacted delta code

After regression areas are identified, a Fix Agent reads the impacted code, understands what changed (the delta), and proposes targeted corrections — not a full rewrite.

Why both indexes? CocoIndex answers semantic queries: "find all places where seat availability is checked" — without knowing the class name. Incremental update on every commit (git hook)

#!/bin/bash
# .git/hooks/post-commit — installed automatically by: graphify hook install

CHANGED=$(git diff --name-only HEAD~1 HEAD -- "*.java")

if [ -n "$CHANGED" ]; then
  # Graphify incremental — re-parses only changed Java files
  nohup graphify update . > /dev/null 2>&1 &

  # CocoIndex incremental — re-embeds only changed AST chunks
  nohup ccc index > /dev/null 2>&1 &

  # code-review-graph incremental
  nohup code-review-graph update > /dev/null 2>&1 &

  echo "[index] Background re-index started for ${#CHANGED} changed files"
fi

Agent design

Four specialised agents — each with one responsibility

🔍
Impact Finder Agent
P1 + P2 — semantic search + blast radius

Given any changed MovieVerse file, queries Graphify graph + CocoIndex semantics + CRG blast-radius to return all affected classes, tests, and API endpoints with risk classification.

graphify query graphify path ccc search CRG blast-radius GRAPH_REPORT.md 
// Example invocation
"TicketingService.bookTickets() has changed.
Find all impacted classes, tests, and endpoints
using the knowledge graph. Output structured JSON."
🧪
Regression Mapper Agent
P2 — maps delta to exact tests from manifest

Takes Impact Finder output, queries test-manifest.json, returns the minimal set of unit and integration tests that must run. Updates the manifest when code structure changes.

test-manifest.json CRG coverage edges mvn test runner 
// Output: tests to run
TicketingServiceTest,
MovieVerseIntegrationTest#bookTickets_success
🗂️
Delta Indexer Agent
P1 — keeps both indexes fresh on every commit

Runs in the post-commit git hook. SHA-256 compares changed Java files against the stored hash manifest. Triggers incremental re-index of only the changed files in Graphify, CocoIndex, and CRG.

git hash-object ccc index graphify update CRG update 
// Reports after each commit
"3 files re-indexed (TicketingService,
TicketingRepository, Booking).
2 test mappings updated in manifest."
📋
PR Reviewer Agent
Orchestrator — runs all agents on every PR

Triggered by GitHub Actions on PR open. Runs Impact Finder → Regression Mapper → Fix Agent → posts a unified comment on the PR with impact report, tests to run, and proposed fixes.

Impact Finder Regression Mapper Fix Agent GitHub PR API

The new agent — P3

Fix Agent — proposes targeted corrections for impacted delta code

★

Delta Fix Agent — reads what changed, proposes surgical fixes

Does not rewrite files · reads only the impacted symbols · proposes minimal diff · references existing patterns

How it works — four-step reasoning loop

1
Read the delta Auto
Agent calls git diff HEAD~1 HEAD -- *.java to get the exact lines that changed. It does not read whole files — only the diff hunks, plus 10 lines of context around each change.
2
Query the graph for impacted symbols Agent
Calls graphify path <changed-class> <impacted-class> for each impacted class identified by the Impact Finder. Retrieves only the specific methods/symbols at risk — not entire files.
3
Analyse and propose fix Agent
For each impacted symbol, the agent reads the current implementation (via CocoIndex semantic search returning the exact method body), understands the breaking change, and proposes a minimal diff that adapts the impacted code to the new contract.
4
Generate fix as unified diff Fix
Output is a standard .patch file — a unified diff that can be reviewed and applied with git apply fix.patch. Human reviews it before anything is committed.

Fix Agent system prompt

# .claude/fix-agent-system-prompt.md

You are a surgical code fix agent for the MovieVerse Spring Boot application.

Architecture:
- 6 domain services: movie, financial, rating, snapshot, recognition, ticketing
- Every service calls MovieService.validateMovieExists() or getMovieById()
- Common layer: ApiResponse<T>, GlobalExceptionHandler, ResourceNotFoundException
- All repositories are in-memory ConcurrentHashMap — no physical database
- Java 21, Spring Boot 3.2, Lombok builders throughout

PROCESS — follow this exactly:
1. Read the git diff to understand what changed (the delta)
2. Call graphify to find impacted classes — do NOT scan all files
3. Use ccc search to retrieve only the impacted method bodies
4. For each impacted method: explain what broke and why
5. Propose a minimal fix as a unified diff
6. Never rewrite an entire file — only the affected methods
7. Reference existing patterns in the codebase (Lombok @Builder, ApiResponse.success(), etc.)
8. If a test needs updating, propose the test fix alongside the code fix

OUTPUT FORMAT:
## Impact summary
[1-2 sentences on what changed and why it breaks downstream]

## Proposed fix: <ClassName>.<methodName>
```diff
- old code
+ new code
```

## Test update required: <TestClass>.<testMethod>
```diff
- old assertion
+ new assertion
```

Worked example — adding a loyaltyPoints field to Booking

Scenario: developer adds loyaltyPointsEarned field to Booking.java and initialises it in TicketingService.bookTickets().

// Developer's change — Booking.java
@Data @Builder
public class Booking {
    // ... existing fields ...
+   private int loyaltyPointsEarned;  // NEW FIELD
}

Fix Agent identifies the blast radius, then proposes:

## Impact summary
Booking.java gained a new field loyaltyPointsEarned. TicketingService.bookTickets()
must populate it. The TicketingServiceTest assertions that verify Booking fields
need updating to expect the new field. No other services are affected since
Booking is only used within the ticketing domain.

## Proposed fix: TicketingService.bookTickets()
--- a/TicketingService.java
+++ b/TicketingService.java
@@ -build section @@
         .confirmationCode(UUID.randomUUID().toString().substring(0,8).toUpperCase())
+        .loyaltyPointsEarned(request.getNumberOfTickets() * 10)
         .build();

## Test update: TicketingServiceTest.bookTickets_success()
--- a/TicketingServiceTest.java
+++ b/TicketingServiceTest.java
@@ -assertion section @@
         assertThat(result.getStatus()).isEqualTo(BookingStatus.CONFIRMED);
+        assertThat(result.getLoyaltyPointsEarned()).isEqualTo(20); // 2 tickets × 10

## Test update: TicketingServiceTest.bookTickets_premiumPricing()
+        assertThat(result.getLoyaltyPointsEarned()).isEqualTo(20); // 2 tickets × 10

CI pipeline

Complete automated pipeline on every PR

1
Developer commits Java change Developer
Any .java file in any of the 6 domain packages. Git post-commit hook fires in background — developer is not blocked. Delta Indexer re-indexes only changed files.
2
Delta Indexer runs Auto · background
SHA-256 compares changed files to .index/hashes.json. Triggers graphify update ., ccc index, and code-review-graph update in parallel background processes. Completes in under 2 seconds for a typical 1–3 file commit.
3
PR opened → GitHub Actions CI
GitHub Actions workflow fires. Checks out code, verifies both indexes are current by comparing current file hashes to stored hashes. Re-indexes if stale.
4
Impact Finder Agent runs Agent
Reads changed files from git diff --name-only. Queries Graphify graph + CRG blast-radius. Returns structured JSON: affected classes, risk level, impacted endpoints, test list.
5
Regression Mapper Agent runs Agent
Takes Impact Finder JSON. Queries test-manifest.json for covering tests. Returns minimal test set: unit tests + integration test methods. Typically 3–11 tests, not the full 35.
6
Fix Agent runs Fix Agent
For HIGH-risk changes: reads delta, queries impacted symbols, proposes a .patch file. Attached to PR comment as a downloadable artifact. Developer reviews and applies with git apply.
7
Targeted tests run CI
Only the tests from the Regression Mapper output run: mvn test -Dtest="...". Pass/fail result determines whether the PR can proceed. Typically completes in under 30 seconds.
8
PR comment posted Auto
PR Reviewer Agent posts unified comment: changed files, blast radius, test results, risk level, and link to the Fix Agent patch file.
9
Merge gate Gate
Blocks merge if: (a) HIGH-risk change with no reviewer approval, (b) targeted tests failed, (c) Impact Finder found blast radius touching MovieService without a full integration test run.

GitHub Actions workflow

# .github/workflows/movieverse-impact.yml
name: MovieVerse Impact Analysis
on:
  pull_request:
    types: [opened, synchronize]
    paths: ['src/**/*.java']

jobs:
  impact-and-fix:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 2 }

      - name: Set up Java 21
        uses: actions/setup-java@v4
        with: { java-version: '21', distribution: 'temurin' }

      - name: Install indexing tools
        run: |
          pip install graphifyy code-review-graph
          pipx install 'cocoindex-code[full]'

      - name: Build + verify indexes current
        run: |
          graphify . --no-llm --force
          ccc index
          code-review-graph build

      - name: Get changed Java files
        run: |
          git diff --name-only HEAD~1 HEAD -- "*.java" > changed.txt
          cat changed.txt

      - name: Run Impact Finder Agent
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p \
            --system-prompt "$(cat .claude/impact-finder-system-prompt.md)" \
            "Analyse impact of: $(cat changed.txt). Use graph + CRG. Output JSON." \
            > impact-report.json

      - name: Run Regression Mapper Agent
        run: |
          claude -p "Given impact: $(cat impact-report.json), query test-manifest.json,
          return exact mvn -Dtest string." > tests-to-run.txt

      - name: Run Fix Agent (HIGH risk only)
        run: |
          RISK=$(cat impact-report.json | python3 -c "import json,sys; print(json.load(sys.stdin).get('risk','LOW'))")
          if [ "$RISK" = "HIGH" ]; then
            claude -p \
              --system-prompt "$(cat .claude/fix-agent-system-prompt.md)" \
              "Delta: $(git diff HEAD~1 HEAD -- '*.java'). Propose fix patch." \
              > fix.patch
          fi

      - name: Run targeted regression tests
        run: |
          mvn test -Dtest="$(cat tests-to-run.txt)" --no-transfer-progress

      - name: Post PR comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const impact = JSON.parse(fs.readFileSync('impact-report.json'));
            const tests = fs.readFileSync('tests-to-run.txt', 'utf8');
            const hasFix = fs.existsSync('fix.patch');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## MovieVerse Impact Report
**Risk:** ${impact.risk} | **Blast radius:** ${impact.affected_files?.length ?? 0} files
**Tests to run:** \`${tests.trim()}\`
${hasFix ? '**Fix Agent:** patch attached — review and apply with \`git apply fix.patch\`' : ''}
`
            });

      - name: Upload fix patch as artifact
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: fix-agent-patch
          path: fix.patch
          if-no-files-found: ignore
Design decisions

Key decisions specific to MovieVerse

MovieService as the god node
Every service in MovieVerse calls MovieService.validateMovieExists(). This is intentional — it means changing Movie.java or MovieService is always a HIGH-risk change and always triggers the full integration test suite. The Impact Finder treats this as a hardcoded rule, not something discovered dynamically.
Fix Agent generates diffs not files
The Fix Agent outputs a .patch file — a unified diff — not modified source files. This preserves human review in the loop: the developer applies it with git apply after inspection. It also means the agent only needs to see the changed method, not the entire file, which reduces token cost significantly.
In-memory repositories need seed validation
MovieVerse uses @PostConstruct seed methods in all repositories. Any change to a model (adding a field, changing a type) can silently break the seed data. The Fix Agent's system prompt explicitly instructs it to check seed methods in repository classes when model files change.
CocoIndex for semantic, Graphify for structural
CocoIndex finds code by meaning ("find all places that validate seat availability") — needed for the Fix Agent to locate the exact method to fix without knowing the class name. Graphify finds structural relationships ("what imports TicketingService?") — needed for the Impact Finder to compute blast radius. Both are needed; they are not interchangeable.
Targeted tests only
The full MovieVerse suite has 35 tests. Running all 35 on every PR takes ~45 seconds. The Regression Mapper limits runs to 3–11 tests per typical change. This makes the CI loop fast enough to iterate — a 10-second run instead of 45 seconds encourages developers to push small commits rather than batching.
Fix Agent runs only on HIGH risk
Running the Fix Agent on every commit would generate noise on trivial changes (renaming a variable, adding a comment). The CI workflow gates it on the Impact Finder's risk classification: LOW and MEDIUM changes get impact + tests only; HIGH risk triggers the full Fix Agent pass.
