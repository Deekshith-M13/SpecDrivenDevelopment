Multi-Repo Impact Analysis
& Regression Agent Architecture

A complete plan for three codebases — React/Next.js frontend, BFF Node.js, and Salesforce B2B Commerce with Apex — where any code change automatically identifies impacted areas, maps regression tests, and updates indexes incrementally.

Problem definition

Three distinct problems — one unified solution

P1

Impact discovery

Developer changes cart calculation Apex class or a shared React component — what else breaks? Find all impacted files, classes, APIs across all three repos before committing.

P2

Regression test mapping

Given a code change, which specific API tests in the regression suite must run? Create the link between changed code and the regression APIs that cover it.

P3

Delta index freshness

When a developer commits, update only what changed using SHA-256 content hashing — not a full re-index. Both Graphify and code-review-graph must stay current incrementally.

Problem 1

Impact discovery — find all affected areas before commit

1

Cross-repo knowledge graph

Graphify merges all three repos into one graph · HTTP_CALLS edges connect React → BFF → Apex

The key capability needed here is a graph that spans all three repos so that changing CartCalculator.cls in Salesforce can be traced forward through the BFF to the React cart component. Graphify's --merge-graphs and cross-repo merge capability enables this.

Step 1 — Build merged cross-repo graph

# Build graph for each repo separately
graphify /repos/react-storefront --no-llm --backend claude-cli
graphify /repos/bff-nodejs --no-llm --backend claude-cli
graphify /repos/salesforce-b2b --no-llm --backend claude-cli

# Merge all three into one cross-repo graph
graphify merge-graphs \
  /repos/react-storefront/graphify-out/graph.json \
  /repos/bff-nodejs/graphify-out/graph.json \
  /repos/salesforce-b2b/graphify-out/graph.json \
  --output /shared/cross-repo-graph.json

# Every node carries a repo tag so you can filter by origin

Step 2 — Build CRG index for each repo

code-review-graph build --path /repos/react-storefront
code-review-graph build --path /repos/bff-nodejs
code-review-graph build --path /repos/salesforce-b2b

Step 3 — Developer queries impact before committing

Inside Claude Code, developer types:

"I am about to change CartCalculator.cls in the Salesforce repo —
specifically the calculateDiscount() method. Using the cross-repo
knowledge graph at /shared/cross-repo-graph.json, find every file,
class, API endpoint, and React component that could be impacted.
Show me: (1) direct dependents, (2) transitive dependents up to
3 hops, (3) which repos each impact is in, (4) risk level."

The Impact Finder Agent (see Agent Design section) handles this query by calling Graphify and CRG tools, traversing the merged graph, and returning a structured impact report.

Problem 2

Regression test mapping — link code changes to API tests

2

Test-to-code manifest + Regression Mapper Agent

Build a bidirectional map: code file → API tests that cover it

The test manifest — the linking layer

The core data structure is a JSON manifest that maps every API test to the code files it exercises. This is built once and updated incrementally on code changes.

// test-manifest.json
{
  "tests": [
    {
      "id": "TC-CART-001",
      "name": "Cart discount calculation",
      "endpoint": "POST /api/v1/cart/calculate",
      "covers_files": [
        "salesforce/CartCalculator.cls",
        "bff/services/cartService.js",
        "bff/routes/cart.js"
      ],
      "sha256_at_map_time": {
        "salesforce/CartCalculator.cls": "a3f92b1c...",
        "bff/services/cartService.js": "7e4d8a12..."
      }
    }
  ]
}

How the manifest is built — three sources

Source 1 — Static analysis: CRG blast-radius traversal from each test file to the production code it imports or calls. This covers unit and integration tests where imports are explicit.

Source 2 — API route matching: Each API test declares the endpoint it hits. The BFF route files expose those endpoints. Graphify's HTTP_CALLS edges connect POST /api/v1/cart/calculate → bff/routes/cart.js → bff/services/cartService.js → Salesforce API call.

Source 3 — AI-generated initial mapping: For the first build, the Regression Mapper Agent reads the test suite and the cross-repo graph and generates the manifest automatically.

# Build initial manifest using Claude Code agent
"Read the test suite at /tests/regression/ and the cross-repo graph
at /shared/cross-repo-graph.json. For each API test, identify which
production code files it exercises based on the endpoint it calls
and the BFF route graph. Output as test-manifest.json"

On every commit — find which tests to run

# CI script: given changed files, find required tests
CHANGED=$(git diff --name-only HEAD~1 HEAD)

# Query manifest: which tests cover these files?
python3 scripts/find_tests.py \
  --manifest test-manifest.json \
  --changed "$CHANGED" \
  --output required-tests.txt

# Run only those tests
jest --testPathPattern="$(cat required-tests.txt | tr '\n' '|')"
# or newman run collection.json --folder "$(cat required-tests.txt)"

Keeping the manifest current

When a file's SHA-256 changes and a code path changes significantly, the manifest entry for tests covering that file is marked stale. The Regression Mapper Agent reruns the mapping for affected tests only — not the full manifest.

Problem 3

Delta index freshness — SHA-256 hash-based incremental update

3

Hash manifest + post-commit hooks

Only files whose SHA-256 changed are re-indexed · Sub-second on typical commits

The hash manifest — one per repo

// .index/hashes.json (per repo)
{
  "salesforce/CartCalculator.cls": "a3f92b1c4d...",
  "bff/services/cartService.js": "7e4d8a1234...",
  "react/components/CartSummary.tsx": "9f2c4b8e...",
  "_last_updated": "2026-06-17T10:42:00Z"
}

The post-commit hook — runs on every commit in every repo

#!/bin/bash
# .git/hooks/post-commit (installed by: graphify hook install)

# 1. Get changed files
CHANGED=$(git diff --name-only HEAD~1 HEAD)

# 2. For each file — compare blob hash to stored hash
NEEDS_REINDEX=()
while read path; do
  CURRENT=$(git hash-object "$path")
  STORED=$(python3 -c "import json; d=json.load(open('.index/hashes.json')); print(d.get('$path',''))")
  if [ "$CURRENT" != "$STORED" ]; then
    NEEDS_REINDEX+=("$path")
  fi
done <<< "$CHANGED"

# 3. Update indexes — only changed files (background, non-blocking)
if [ ${#NEEDS_REINDEX[@]} -gt 0 ]; then
  nohup graphify update . > /dev/null 2>&1 &
  nohup code-review-graph update > /dev/null 2>&1 &
  echo "[index] Updating ${#NEEDS_REINDEX[@]} changed files in background"
fi

Branch switch — full rebuild

#!/bin/bash
# .git/hooks/post-checkout
# Runs when switching branches — do a full rebuild
graphify . --no-llm --force &
code-review-graph build &
echo "[index] Full rebuild triggered for branch switch"

Both tools run in the background. The developer's git commit returns immediately. Index updates happen in a nohup background process. The developer is never blocked waiting for re-indexing.
Agent design

Four specialised agents — each with a single responsibility

Each agent is a Claude Code session with a specific system prompt, a specific set of MCP tools, and a specific output format. Agents can be invoked manually by developers or automatically by CI.

🔍
Impact Finder Agent
Problem 1 — blast radius across all three repos

Given a changed file or function, traverses the cross-repo graph and returns all impacted files, classes, endpoints, and components with risk levels.

graphify query graphify path CRG blast-radius cross-repo-graph.json 
// Invocation in Claude Code
"Run impact analysis for CartCalculator.cls
→ calculateDiscount(). Use the cross-repo graph.
Output: affected files per repo, impacted endpoints,
risk level, recommended tests."
🧪
Regression Mapper Agent
Problem 2 — map code changes to API tests

Given changed files from the Impact Finder, queries the test manifest and returns the exact regression tests that must run. Updates the manifest when coverage changes.

test-manifest.json CRG coverage edges BFF route graph API test suite 
// Auto-triggered from Impact Finder output
"Given these changed files: [list from P1],
find all regression tests from test-manifest.json
that cover them. Return test IDs and commands
to run them."
🗂️
Delta Indexer Agent
Problem 3 — keep indexes fresh after every commit

Runs in CI on every PR. Computes which files changed via SHA-256 hash comparison, triggers incremental re-index of only those files across all three repos, updates the test manifest.

git diff --name-only git hash-object graphify update CRG update 
// Runs automatically in CI post-commit
"Compare current file hashes to .index/hashes.json.
Re-index only changed files. Update test-manifest.json
for files whose hash changed. Report: N files reindexed,
M test mappings updated."
📋
PR Review Agent
Orchestrator — combines all three agents per PR

Triggered on every GitHub PR. Runs Impact Finder → Regression Mapper → Delta Indexer in sequence and posts a unified impact + test report as a PR comment.

Impact Finder Regression Mapper Delta Indexer GitHub API 
// Posted as GitHub PR comment

Blast radius: 7 files across 3 repos

Risk: HIGH — touches checkout flow
Index updated: 3 files reindexed

Agent system prompts

Each agent's behaviour is defined in a dedicated CLAUDE.md or --system-prompt flag in the CI step. The Impact Finder Agent system prompt specifically:

# Impact Finder Agent — system prompt

You are an impact analysis agent for a three-repo commerce platform:
- React/Next.js storefront (components, hooks, pages)
- Node.js BFF (routes, services, API aggregation)
- Salesforce B2B with Apex (CartCalculator, triggers, flows)

Cross-repo knowledge graph: /shared/cross-repo-graph.json
Code-review-graph is available via MCP tools.

MANDATORY PROCESS:
1. Read cross-repo-graph.json to understand the dependency structure
2. Use graphify query to find direct dependents of the changed symbol
3. Use CRG blast-radius to find transitive dependents (max 3 hops)
4. Classify each dependent: DIRECT / TRANSITIVE / POTENTIAL
5. Identify which API endpoints are in the blast radius
6. Assign risk: HIGH (checkout/payment), MEDIUM (cart/product), LOW (display only)
7. Output the structured impact report format

Never read raw files to discover structure. Always use graph tools first.
CI pipeline

Complete automated pipeline on every PR

1
Developer commits code Developer
Any file in any of the three repos. Git post-commit hook fires immediately in background — developer is not blocked.
2
Delta Indexer runs Auto
SHA-256 hash comparison against .index/hashes.json. Only changed files re-indexed by Graphify and CRG. Typically completes in under 2 seconds for a normal commit.
3
PR opened → GitHub Actions triggers CI
GitHub Actions workflow fires on PR open and push. Checks out all three repos, confirms indexes are current, runs the PR Review Agent.
4
Impact Finder Agent runs Agent
Reads changed files from git diff. Queries cross-repo graph. Returns blast radius: direct dependents, transitive dependents up to 3 hops, impacted API endpoints, risk level.
5
Regression Mapper Agent runs Agent
Takes Impact Finder output. Queries test-manifest.json for all tests covering impacted files. Returns exact test IDs and run commands.
6
Targeted regression suite runs CI
Only the tests identified by the Regression Mapper run — not the full suite. For a cart calculation change: typically 3–8 tests, not 200. Completes in seconds not minutes.
7
PR comment posted Auto
PR Review Agent posts the unified impact report as a GitHub PR comment. Reviewer sees: changed files, blast radius, tests run, pass/fail, risk level — before approving.
8
Merge gate Gate
PR cannot merge if: (a) HIGH risk with no reviewer approval, (b) regression tests failed, (c) blast radius touches checkout/payment with no QA sign-off.

GitHub Actions workflow

# .github/workflows/impact-analysis.yml
name: Impact analysis and regression mapping
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  impact-analysis:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }

      - name: Install tools
        run: |
          pip install graphifyy code-review-graph
          npm install -g @anthropic-ai/claude-code

      - name: Get changed files
        run: |
          git diff --name-only origin/main...HEAD > changed_files.txt

      - name: Run Impact Finder Agent
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p \
            --system-prompt "$(cat .claude/impact-finder-system-prompt.md)" \
            "Analyse impact of these changed files: $(cat changed_files.txt).
             Use the cross-repo graph. Output structured JSON." \
            > impact-report.json

      - name: Run Regression Mapper Agent
        run: |
          claude -p \
            "Given this impact report: $(cat impact-report.json),
             query test-manifest.json and return tests to run." \
            > tests-to-run.txt

      - name: Run targeted regression tests
        run: |
          npm run test:regression -- --filter="$(cat tests-to-run.txt)"

      - name: Post PR comment
        uses: actions/github-script@v7
        with:
          script: |
            const report = require('./impact-report.json');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: formatImpactReport(report)
            });
Design decisions

Key decisions and why

Cross-repo merged graph
A developer changing Apex code cannot see React impact without a graph that spans all three repos. Graphify's merge-graphs command produces a single traversable graph with repo tags. This is the only way to answer "what React components are affected by this Apex change?"
SHA-256 over timestamps
File timestamps change on git operations that do not change content (checkout, merge, rebase). SHA-256 content hashing is the only reliable signal that file content actually changed. Both Graphify and CRG use content hashing — this blueprint preserves that.
Test manifest as link layer
No existing tool directly maps API regression tests to source files across repos and languages. The test-manifest.json is a custom layer built once by the Regression Mapper Agent and maintained incrementally. It is the key piece that solves Problem 2.
Agents not monolithic scripts
Splitting into four agents (Impact Finder, Regression Mapper, Delta Indexer, PR Reviewer) means each can be invoked independently, improved independently, and have clear system prompts. A monolithic script would conflate concerns and be harder to maintain.
Background indexing
Both Graphify and CRG re-index in a nohup background process so git commit returns immediately. Developer experience is zero-latency — the index catches up within seconds without blocking the workflow.
Targeted tests not full suite
Running 200 regression tests on every PR is slow and creates alert fatigue. The Regression Mapper runs only tests linked to the blast radius — typically 3–15 tests. This makes CI fast enough to be useful rather than a bottleneck.
